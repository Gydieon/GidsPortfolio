Core Functionality Testing

Test basic osTicket features to ensure the system is functioning correctly. These are the fundamental operations every helpdesk performs daily.

1

Ticket Creation & Submission

Verify that clients can create tickets successfully

Test Scenario 1.1: Client Portal Ticket Creation
  1. Open client portal: http://osticket.local/
  2. Click "Open a New Ticket"
  3. Fill form with test data:
    • Name: "John Test"
    • Email: "john@example.com"
    • Subject: "Network connection issue"
    • Priority: "High"
    • Message: "I cannot connect to the office network"
  4. Submit the ticket
  5. Note the ticket number displayed (e.g., #ABC123)
  6. Verify confirmation message appears

Expected Results

  • ✓ Ticket form accepts all input data
  • ✓ Ticket created with unique ID
  • ✓ Confirmation page shows ticket number and reference info
  • ✓ Customer receives confirmation email immediately
  • ✓ Ticket appears in admin dashboard
Test Scenario 1.2: Admin Ticket View & Response
  1. Log into admin panel: http://osticket.local/admin/
  2. Navigate to Tickets → Open
  3. Locate the test ticket created above
  4. Click to open the ticket details
  5. Verify all information is displayed correctly
  6. Change status from "Open" to "In Progress"
  7. Add a response: "We're investigating this issue"
  8. Click "Post Reply"

Expected Results

  • ✓ Ticket displays in admin list with correct info
  • ✓ Full conversation history visible
  • ✓ Status can be changed successfully
  • ✓ Response can be added without errors
  • ✓ Customer receives response via email
  • ✓ Response timestamp recorded correctly

Email System Testing

Verify email integration is working properly in both directions—sending notifications and receiving customer emails.

2

Outgoing Email (SMTP) Testing

Verify osTicket can send emails successfully

Test Scenario 2.1: Test Email Settings
  1. Log into admin panel
  2. Go to Settings → Email → Outgoing Email (SMTP)
  3. Click "Send Test Email"
  4. Enter your email address in the test field
  5. Wait 1-2 minutes for the test email
  6. Check your inbox (and spam folder)
  7. Verify the test email was received

Expected Results

  • ✓ Test email button responds without errors
  • ✓ Test email received within 2 minutes
  • ✓ Email header shows correct from address
  • ✓ Email content is clear and readable
  • ✓ No SPF/DKIM warnings in email headers
Test Scenario 2.2: Notification Email Delivery
  1. Create a new test ticket (refer to section 1)
  2. Use your own email (e.g., you@example.com) as submitter
  3. Check your inbox immediately
  4. Verify you receive the acknowledgment email
  5. Add a response to the ticket from admin panel
  6. Check inbox again for the response notification

Expected Results

  • ✓ Acknowledgment email sent within 30 seconds
  • ✓ Email contains ticket number and reference link
  • ✓ Response email sent when staff replies
  • ✓ Email formatting is correct (no garbled text)
  • ✓ All email timestamps are accurate
3

Incoming Email (IMAP) Testing

Verify osTicket can receive emails from customers

Test Scenario 2.3: Email to Ticket Conversion
  1. From your email client, compose a new email
  2. Send to the support email configured in osTicket (e.g., support@yourdomain.com)
  3. Subject: "Test email to ticket"
  4. Body: "This is a test to see if email creates a ticket"
  5. Wait 5 minutes (or click "Fetch Now" in Incoming Email settings)
  6. Check osTicket admin dashboard for new ticket

Expected Results

  • ✓ New ticket created from incoming email
  • ✓ Ticket subject matches email subject
  • ✓ Ticket body contains email message
  • ✓ Email sender's info captured correctly
  • ✓ Attachments (if any) are included

Routing & Automation Testing

Verify that routing rules and automation rules work correctly to distribute tickets and automate actions.

4

Email Routing Rules

Test that tickets are routed to correct departments

Test Scenario 4.1: Department-Based Routing
  1. Send an email to: network-support@yourdomain.com
  2. Subject: "VPN not working"
  3. Body: "Cannot connect to corporate VPN"
  4. Wait for email fetch (or click Fetch Now)
  5. Check if ticket was created in "Network & Security" department
  6. Verify ticket priority is set to High
  7. Verify SLA is set to "Critical"

Expected Results

  • ✓ Ticket created in correct department
  • ✓ Department manager can see the ticket
  • ✓ Correct priority assigned automatically
  • ✓ Correct SLA applied
  • ✓ Department email signature included
5

Automation Rules

  5. Check that email contains helpful information
Test Scenario 4.3: Auto-Escalation
  1. If configured, wait for old tickets to be escalated
  2. Check priority of aged tickets
  3. Verify managers are notified of escalations
  4. Confirm ticket status reflects escalation

SLA Validation

Verify that SLA timers start, pause, and track correctly based on ticket status and time.

6

SLA Timer Functionality

Test SLA response and resolution time tracking

Test Scenario 6.1: SLA Timer Display
  1. Create a Critical priority ticket
  2. Open ticket details in admin panel
  3. Look for SLA timer display on ticket
  4. Verify it shows time remaining for response
  5. Note the response deadline (e.g., 2 hours from creation)
  6. Note the resolution deadline (e.g., 4 hours from creation)

Expected Results

  • ✓ SLA plan applied based on priority
  • ✓ Response deadline clearly displayed
  • ✓ Resolution deadline clearly displayed
  • ✓ Timers update correctly as time passes
  • ✓ Timer color indicates urgency (red when close to deadline)
Test Scenario 6.2: SLA Pause on Wait
  1. Add a response to the ticket
  2. Change status to "Awaiting Customer" (if this pauses SLA)
  3. Note the SLA timer value
  4. Wait 1 minute
  5. Check if SLA timer is still the same (paused)
  6. Change status back to "Open"
  7. Verify timer resumes counting

User Roles & Access Control

Staff Role Permissions

Test different staff roles and their permissions

  2. Log out as Admin
  3. Log in as the Agent
  4. Verify agent can see their department's tickets
  5. Verify agent CANNOT see other departments' tickets
  6. Verify agent CAN respond to tickets
  7. Verify agent CANNOT access Settings
  8. Verify agent CANNOT create new staff accounts
Test Scenario 7.2: Department Manager Access

  2. Log in as Manager
  3. Verify manager can see all department tickets
  4. Verify manager can view reports
  5. Verify manager CANNOT access system Settings
  6. Verify manager CANNOT manage other departments

Security Verification

Verify that security measures are in place to protect the helpdesk system and customer data.

9

Access Control & Encryption

Test security features and data protection

Test Scenario 9.1: Login Security
  1. Try to access /admin without logging in
  2. Verify you're redirected to login page
  3. Try invalid credentials (wrong password)
  4. Verify login fails with appropriate message
  5. Try valid credentials
  6. Verify login succeeds
  7. Check if session cookie is secure (HTTPS)
Test Scenario 9.2: Data Privacy
  1. Log in as an Agent
  2. Try to access Settings (should be denied)
  3. Try to view another department's tickets
  4. Verify access is denied if you don't have permission
  5. Logout and verify session is cleared

Security Checklist

  • ✓ Login required to access admin panel
  • ✓ Invalid passwords rejected
  • ✓ Session timeout active (auto-logout after inactivity)
  • ✓ Staff cannot view other departments' tickets (if configured)
  • ✓ Setup directory removed or protected
  • ✓ Configuration file permissions set correctly (644)
  • ✓ Database credentials not visible in client portal
